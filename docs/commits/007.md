# 007 — feat(outbox): 정산 이벤트 transactional outbox publisher

## 개요
outbox 행을 Kafka로 흘리는 발행 측. betting/wallet의 outbox를 미러링(검증된 패턴).

## 작업 내역
- **`OutboxPublisher`**: `@Scheduled(fixedDelay 1s)` + `@Transactional`. 미발행 행 배치(100) fetch → 각각 Kafka send + ack 대기 → 성공 시 `markPublished`. ack 실패 행은 미발행으로 남아 다음 tick 재시도 — **retry-until-acked = at-least-once**.
- **`SettlementEventFactory`**: `BetSettled`/`BetVoided` Avro를 빌드해 `OutboxEvent`로. partition key=eventId.
  - `BetSettled`(betId, userId, eventId, result, stake, payout, settledAt) — WON/LOST/PUSH/VOID 결과.
  - `BetVoided`(betId, userId, eventId, reason, refund, voidedAt) — 경기 취소·연기 전액 환불.
  - `SettlementResult`(domain) → `SettlementResultAvro`(생성 enum)를 `valueOf(name())`로 미러.
- **`KafkaConfig`**: `(String, byte[])` idempotent producer + `KafkaTemplate`. payload는 Avro raw bytes(Schema Registry 없음, ADR-0014).
- **`AvroSerializer`**: SpecificRecord ↔ byte[]. betting은 발행만(AvroSerializer), risk는 consume만(AvroCodec) — settlement은 둘 다 하므로 두 헬퍼가 공존(각 단일 책임, 기존 repo 관례와 동일).

```java
@Scheduled(fixedDelayString = "${settlement.outbox.poll-interval-ms:1000}")
@Transactional
public void publishPending() {
  for (OutboxEvent e : repo.findUnpublished(PageRequest.of(0, 100)))
    if (sendBlocking(e)) e.markPublished(clock.instant());   // ack 못 받으면 다음 tick 재시도
}
```

## 결과
정산 결과를 발행할 인프라 완성. 아직 호출자(SettlementService)는 없음.

## 요약
- 1초 cadence 스케줄 publisher, at-least-once.
- BetSettled/BetVoided 팩토리, eventId partition key.

## 다음 작업
wallet credit client → 둘을 묶는 SettlementService.

## 핵심 확인
`SettlementServiceIntegrationTest`가 outbox 행을 디코드해 검증(커밋 010).

## 기억/설명 Level
- **L1**: publisher cadence, 두 이벤트 종류.
- **L2**: retry-until-acked가 왜 at-least-once인가, idempotent producer.
- **L3**: 발행 측 at-least-once를 consume 측 멱등이 어떻게 보완하는가(전체 그림).
