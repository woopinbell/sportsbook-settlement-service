# 009 — feat(settlement): resolver+payout+outbox 오케스트레이션

## 개요
정산의 심장. 한 베팅을 한 이벤트에 대해 정산한다(`SettlementService`). 설계 제약: **동기 wallet HTTP를 열린 DB 트랜잭션 안에서 호출하지 않는다**(betting의 교훈).

## 핵심 설계 — 2-phase + row lock + 재가드
```
settleOnEvent(betId, eventId, resolution):
  ① prepare  (tx + row lock): event의 selection outcome stamp.
              다른 이벤트가 아직 PENDING이면 stamp만 커밋하고 중단(멀티이벤트 대기).
              전부 resolve됐으면 resolver로 결과/payout 계산.
  ② credit   (tx 밖): WON→ stake 반환(USER_LOCKED) + 이익(HOUSE_POOL),
              PUSH/VOID→ 환불, LOST→ 무호출. 각 leg는 betId 파생 멱등 키.
  ③ finalize (tx): PENDING **재가드** → SETTLED 전이 + outbox 행.
```
- **왜 2-phase인가**: 단순 `@Transactional` 한 메서드면 wallet HTTP가 DB 트랜잭션을 붙잡는다(connection pool 압박). prepare/finalize를 `TransactionTemplate`로 쪼개 그 사이에 credit을 끼운다.
- **멀티이벤트 multi**: 한 베팅이 event A,B에 걸쳐 있으면 A의 MatchResult가 와도 B 미해결 → prepare가 A만 stamp하고 중단(`allSelectionsResolved()` false). B 도착 시 전부 resolve → 정산. **모든 이벤트의 결과가 모여야 정산**.
- **동시성/멱등**: ③의 `isPending()` 재가드 + betId 파생 멱등 키. 재전달·replay·멀티이벤트 동시 도착에도 중복 정산/중복 payout 없음. prepare의 row lock(`findForUpdateById`)이 한 멀티의 이벤트들을 직렬화해 "전부 resolve됐나" 판정을 일관되게.
  - `FOR UPDATE` + `JOIN FETCH` 동시 사용은 Postgres가 거부(outer join의 nullable 쪽) → 락은 fetch 없이, selection은 같은 트랜잭션에서 lazy 로드.

## 작업 내역
- `Bet`에 전이 메서드(`recordSettled`/`recordVoided`, PENDING 가드) + `applyEventOutcomes`(event별 selection stamp) 추가.
- `BetSelection.recordOutcome` (write-once, package-private).
- `SettlementResolver.lineCount` 추가 — committed stake(= unit×lineCount) 계산용(WON의 stake 반환액). betting `SystemBetCalculator.lineCount`와 대칭.
- `EventResolution`(mode + selectionId→결과 맵, Avro-free) + `findForUpdateById`(PESSIMISTIC_WRITE)·`findPendingByEvent` repo 쿼리 + `TransactionTemplate` 빈.

## 결과
정산 전 과정이 한 컴포넌트로. 다음 커밋의 WireMock 통합 테스트로 증명.

## 요약
- 2-phase(prepare/credit/finalize), HTTP를 tx 밖으로.
- 멀티이벤트는 전부 resolve 시 1회 정산.
- isPending 재가드 + 멱등 키 = 중복 0.
