# 002 — feat(readmodel): BetPlacedRequested로 read model 구축

## 개요
`bet.placed.v1`(betting이 실제 publish하는 토픽)을 consume해 정산용 bet 스냅샷을 자체 구축한다. betting DB를 직접 읽지 않으므로 두 서비스가 느슨하게 결합된다(ADR-0006/0017).

## 작업 순서
1. JPA 엔티티: `Bet`(집계 루트) + `BetSelection`(자식), `SlipKind`/`SettlementStatus` enum, `EmbeddedMoney`/`UuidV7` 인프라.
2. `AvroCodec`(byte[]↔SpecificRecord 디코드).
3. `BetReadModelWriter` — betId 기준 멱등 upsert.
4. `BetPlacedConsumer` — `@KafkaListener` 디코드 → Avro-free `BetPlacement`로 매핑 → writer.

## 작업 내역
- **Bet 집계**: 스냅샷(slip/stake/selection)과 정산 상태(status/result/payout)를 한 엔티티에. `betSlipType()`이 저장된 `SlipKind`+K/N으로 shared-protocol의 `BetSlipType` sealed interface를 복원 → resolver가 그걸 소비.
  - C++ 관점: `sealed interface BetSlipType permits Single, Multiple, System`은 닫힌 타입 계층(Rust enum / C++ `std::variant`와 유사). DB엔 평평한 `SlipKind` enum + nullable K/N으로 저장하고, 도메인 경계에서 sealed 타입으로 되살린다.
- **Avro-free 경계**: consumer가 `BetPlacedRequested`(Avro 생성 클래스)를 디코드한 뒤 즉시 `BetPlacement`(순수 record)로 변환. Avro 타입이 persistence 레이어로 새지 않게 한다.
- **멱등 upsert**: `existsById(betId)`면 no-op. betting이 userId 키로 publish → 한 bet의 이벤트는 한 partition에 단일 스레드 처리되므로 자기 자신과 race하지 않고, `bet_id` PK가 최후 방어선.
```java
@KafkaListener(topics="${settlement.topics.bet-placed}", groupId="settlement.bet-placed")
public void onBetPlaced(@Payload byte[] payload, @Header(...) String key, Acknowledgment ack) {
  BetPlacedRequested e = AvroCodec.decode(payload, BetPlacedRequested.class);
  boolean created = writer.record(toPlacement(e));
  ack.acknowledge();              // 쓰기 커밋 후에만 ack → 실패 시 재전달
}
```
- **manual ack**: `manual_immediate` 모드라 처리 성공 후 `ack.acknowledge()`를 명시 호출해야 offset이 전진한다. C++엔 없는 Kafka 개념 — at-least-once 보장의 핵심.

## 결과
이벤트 → PENDING bet 행 + selection들. 멱등.

## 요약
- Bet/BetSelection 엔티티 + 멱등 read-model writer + Avro consumer.
- Avro는 event 레이어에 가두고 domain은 순수 value object만.
