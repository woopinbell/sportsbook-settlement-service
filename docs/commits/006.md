# 006 — build(flyway): V3 transactional outbox 스키마

## 개요
정산 결과(BetSettled/BetVoided)를 Kafka로 내보내되, **상태 전이와 이벤트 발행이 갈라지지 않게** transactional outbox(ADR-0006)를 쓴다. 그 저장소 테이블.

> 마이그레이션 번호가 V1 → V3으로 점프한다(V2 없음). 내용 기준 명명을 쓰다 번호에 갭이 생겼고, Flyway는 버전 갭을 허용하며 순서대로 적용하므로 무해하다. 굳이 renumber하지 않는다(이미 push된 마이그레이션 rename은 checksum 혼란).

## 작업 내역
```sql
CREATE TABLE outbox_event (
  event_id PRIMARY KEY, topic, partition_key, schema_name,
  payload BYTEA,                                  -- Avro binary (no Schema Registry, ADR-0014)
  created_at, published_at);                       -- published_at NULL = 미발행
CREATE INDEX ix_outbox_unpublished ON outbox_event(created_at) WHERE published_at IS NULL;
```
- **outbox 패턴 원리**: bet의 SETTLED/VOIDED 전이와 outbox_event INSERT를 **한 DB 트랜잭션**에. 둘 다 커밋되거나 둘 다 롤백 → "정산했는데 이벤트 유실" 또는 "이벤트는 갔는데 정산 안 됨"이 구조적으로 불가능. 별도 publisher가 미발행 행을 Kafka로 흘리고 ack 시 `published_at` stamp.
- **partition_key = eventId** (ADR-0006): 한 경기의 정산 스트림이 같은 partition → 순서 보장. betting은 userId 키였지만 정산은 경기 단위.
- **partial index** `WHERE published_at IS NULL`: publisher가 "미발행 행 오래된 순"만 스캔. 발행 완료 행이 쌓여도 스캔이 작다.

## 결과
스키마만. betting/wallet의 outbox와 동일 구조(재사용 검증된 패턴).

## 요약
- outbox_event 테이블, partition key=eventId, 미발행 partial index.
- 상태 전이와 이벤트 발행의 원자성 보장 기반.

## 기억/설명 Level
- **L1**: outbox_event 컬럼.
- **L2 (외워서 설명 권장)**: **outbox 패턴** — 상태+이벤트를 한 트랜잭션에 써서 dual-write 문제를 푼다. partition key=eventId 이유.
- **L3**: Flyway 버전 갭이 왜 무해한가.
