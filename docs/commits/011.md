# 011 — feat(trigger): MatchResult/EventLifecycle consumer

## 개요
Kafka 토픽을 SettlementService로 잇는 얇은 어댑터 2개 + fan-out 트리거.

## 작업 내역
- **`MatchResultConsumer`** (`match.result`): Avro `MatchResult` 디코드 → `EventResolution`으로 매핑(finalStatus→mode, **resultDetail→selectionId별 결과 계약**) → `SettlementTrigger.onMatchResult`.
  - **판정 입력 계약**: `resultDetail`(map<string,string>)을 `selectionId → {WON/LOST/PUSH/VOID}`로 파싱. settlement은 score를 재해석하지 않는다(경계: 결과 판정 안 함). 잘못된 값(bad UUID/unknown result)은 throw → 비-ack → 재시도/DLQ(silent 오정산 방지).
- **`EventLifecycleConsumer`** (`event.lifecycle`): CANCELLED→`EVENT_CANCELLED` void, POSTPONED→`EVENT_POSTPONED` void(ADR-0012). FINISHED는 정보성(정산은 더 풍부한 MatchResult가 구동), SCHEDULED/IN_PLAY 무시.
- **`SettlementTrigger`**: `findPendingByEvent(eventId)` → 각 베팅에 settleOnEvent / voidOnEvent fan-out. per-bet 작업은 멱등 SettlementService라 redelivery·replay 안전. 베팅이 아직 read model에 없으면(BetPlacedRequested 미consume) 그냥 안 잡힘 → 후속 replay로 정산.

```java
EventResolution resolution = new EventResolution(mode(e.getFinalStatus()), outcomes(e));
trigger.onMatchResult(eventId, resolution, e.getSettledAt());
ack.acknowledge();
```

## 결과
이벤트 → 정산 자동 구동. 다음 커밋의 통합 테스트로 end-to-end 증명.

## 요약
- 두 consumer(얇은 Avro 어댑터) + fan-out 트리거.
- resultDetail 계약으로 selection별 결과 수신, score 재해석 안 함.
- CANCELLED/POSTPONED만 void, FINISHED 정보성.
