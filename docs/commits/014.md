# 014 — feat(settlement): match result 영속화 + replay (+ 동시성 증명)

## 개요
MatchResult 트리거가 fan-out 전에 결과를 `match_result`로 영속화하고, `SettlementTrigger.replay(eventId)`로 stored 결과에서 PENDING 베팅을 재정산한다(admin replay + late-bet catch-up).

## 작업 내역
- **영속화 + fan-out**: `onMatchResult`가 `MatchResultRecord` save 후 `fanOutSettle`. 영속화가 먼저라 fan-out이 일부 실패해도 결과는 남고, 재전달 시 재정산.
- **replay**: stored `MatchResultRecord` → `EventResolution` 재구성 → PENDING 베팅 재정산. 없으면 false(컨트롤러가 404).
- **mode enum을 domain으로**: `MatchResultRecord`(domain 엔티티)가 `EventResolution.Mode`(service)를 참조하면 **패키지 순환**(domain↔service). `MatchOutcomeMode`를 domain으로 올려 둘이 공유(service→domain 단방향). EventResolution 리팩터 + 테스트 1곳 수정.
  - C++ 관점: Java는 패키지 간 순환 의존을 컴파일은 허용하나 설계 악취. 헤더 순환 의존(`#include` 사이클)을 forward declaration으로 푸는 것과 비슷하게, 공용 타입을 하위 레이어로 내린다.
- **동시성 증명**(settlement IT에 추가): 16스레드가 같은 betId를 동시 정산 → **정확히 1회 정산**(outbox 1행). `isPending` 재가드가 보증하고, 모든 credit이 같은 betId 파생 키라 실제 wallet은 단일 payout으로 dedup → **중복 payout 0**(Level 3 정합성).

```java
public boolean replay(UUID eventId) {
  return matchResults.findById(eventId)
     .map(r -> { fanOutSettle(eventId, new EventResolution(r.mode(), r.selectionOutcomes())); return true; })
     .orElse(false);
}
```

## 결과
replay 가능 + 동시 재정산 안전 증명. mode enum 순환 해소.

## 요약
- match_result 영속화 → replay/late-bet 기반.
- MatchOutcomeMode를 domain으로(순환 회피).
- 16스레드 동시 정산 → 1회(중복 payout 0).
