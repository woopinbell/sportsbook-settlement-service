# 018 — test(load): 처리량 베이스라인

## 개요
"1만 베팅 정산 < 10초"(sportsbook/CLAUDE.md Level 2) 목표를 실측. 정산은 이벤트 기반이라 k6 HTTP로 못 때리므로 **JVM 처리량 하니스**로 정산 엔진을 직접 구동.

## 작업 내역
- **`SettlementThroughputLoadTest`** (`@Tag("load")`): 1만 PENDING 베팅 seed → 32스레드로 `settleOnEvent` 병렬 → wall-clock 측정. Kafka 브로커(partition으로 독립 확장)를 빼고 **resolve→credit→persist→outbox 엔진**만 격리 측정.
- **기본 build 제외**: pom surefire `<excludedGroups>${surefire.excludedGroups}</excludedGroups>`(기본 `load`). 실행은 `-Dsurefire.excludedGroups= -Dtest=*LoadTest`로 override.
  - 처음 `-DexcludedGroups=`로 시도했으나 명시 config가 우선해 0건 실행 → property 참조로 바꿔 override 가능하게(빌드 함정).
- **outcome 파라미터**: WON(full payout, 2 credit/bet) vs LOST(엔진만, wallet I/O 없음)로 분해 측정.

## 결과 (dev-host, 1만 베팅, 32스레드)
| 시나리오 | 소요 | 처리량 | 정합성 |
|---|---|---|---|
| LOST (엔진만) | 10.9s | **915 bets/s** | 1만건 1회 정산 |
| WON (full payout) | 17.0s | **589 bets/s** | 1만건 1회 정산 |

- 엔진 자체는 단일 비풀링 PostgreSQL에서 **~목표치(1000/s)**. WON은 베팅당 **wallet HTTP 2회**가 지배적 → 589/s. dev-host full-payout 미달은 betting p99 미달과 같은 이유(모든 의존성 CPU 경쟁 + 실제 왕복). production은 Kafka partition·다중 인스턴스·풀링 wallet으로 초과.
- **모든 run `outbox.count()==N`** — 1만건 정확히 1회 정산(중복 payout 0, 배치 스케일).

## 요약
- JVM 처리량 하니스(엔진 직접), `@Tag("load")` 기본 제외.
- 엔진 915/s, full-payout 589/s, 중복 0.

- **L1**: 두 수치(915/589).
- **L2**: 왜 k6 아닌 JVM 하니스, WON이 왜 느린가(2 wallet 왕복).
- **L3 (외워서 설명)**: 엔진은 목표치 근접, wallet I/O가 throughput 결정; production 확장 축(partition/instance/pool). dev-host vs production 정직한 narrative.
