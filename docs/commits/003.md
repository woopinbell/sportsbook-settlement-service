# 003 — test(readmodel): consume + 멱등 증명

## 개요
read model의 핵심 불변식을 증명: `BetPlacedRequested`를 consume하면 PENDING bet 행이 생기고, **같은 이벤트 재전달은 멱등**이다(중복 행 없음).

## 작업 순서
1. `@SpringBootTest` + Testcontainers PostgreSQL + `@EmbeddedKafka`.
2. 토픽에 Avro 이벤트 produce → bet 행 생성 await.
3. 같은 이벤트 재전달 → 행 수 1 유지를 **윈도우 동안** 검증.

## 작업 내역
- **인프라 조합**: Testcontainers로 실제 PostgreSQL을 띄우고(`@DynamicPropertySource`로 jdbc url 주입), `@EmbeddedKafka`로 in-JVM 브로커. 실제 직렬화/역직렬화/DB 왕복을 다 거치는 통합 테스트.
- **멱등 검증의 함정**: 두 번째 전달 처리가 비동기라 "방금 추가 안 됐다"를 단언하기 어렵다. Awaitility의 `during(2s).atMost(5s)`로 **윈도우 내내** `count()==1`이 유지되는지 확인 — 일시적 double-insert가 있었다면 잡힌다.
```java
await().during(ofSeconds(2)).atMost(ofSeconds(5))
  .untilAsserted(() -> {
     assertThat(bets.count()).isEqualTo(1L);
     assertThat(bets.findWithSelectionsById(betId).orElseThrow().selections()).hasSize(2);
  });
```
- **lazy 컬렉션 함정**: 트랜잭션 밖에서 `bet.selections()`(LAZY) 접근 시 `LazyInitializationException`. fetch-join 쿼리 `findWithSelectionsById`를 추가해 한 방에 로드(이후 resolver도 사용).

## 결과
2 테스트 green (생성 / 멱등).

## 요약
- Testcontainers PG + EmbeddedKafka 통합.
- `during` 윈도우로 멱등을 견고하게 단언.
- fetch-join으로 lazy 회피.
