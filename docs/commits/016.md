# 016 — feat(api): 상태/replay/void REST + late-settlement 윈도우

## 개요
정산 자체는 이벤트 기반이지만, 조회 + 운영 액션 2개를 위한 내부 REST를 연다(admin-api가 인증된 front door). late settlement 24h 윈도우(ADR-0012)도 여기서.

## 작업 내역
- **REST** (`/internal/v1/settlements`):
  - `GET /{betId}` — `SettlementView`(status/result/payout).
  - `POST /replay/{eventId}` — stored 결과로 PENDING 재정산(DLQ drain·late correction 후 운영 액션). 결과 없으면 404.
  - `POST /void/{betId}` — 수동 void + 환불(`voidByAdmin`, reason ADMIN_VOID). 비-PENDING이면 409, 미존재 404.
- **RFC 7807** (`SettlementExceptionHandler`): 에러를 `application/problem+json`으로(shared-protocol `ProblemDetail`). NOT_FOUND/CONFLICT는 shared `ErrorCode` 카탈로그에 없어 inline 생성(카탈로그는 betting 거절 코드만 seed), 바인딩 실패→VALIDATION_FAILED.
- **late settlement 윈도우**: `SettlementTrigger.onMatchResult`가 이미 materialized된 이벤트에 대한 *정정*은 `correction-window`(기본 24h, 원 settledAt 기준) 내에서만 자동 적용. 이후 정정은 consumer 경로에서 무시 → admin replay 필요. 첫 결과는 항상 수용.
- `voidByAdmin`: `prepareVoid`가 베팅 첫 이벤트를 partition key로 캡처(BetVoided 순서용), reason ADMIN_VOID로 finalize.

## 결과
운영 조회·액션 endpoint + 일관된 RFC 7807. late-settlement 정책 강제.

## 요약
- GET status / POST replay / POST void + RFC 7807.
- 24h 자동 정정, 이후 admin replay(ADR-0012).
