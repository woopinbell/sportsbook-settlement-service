# 008 — feat(payout): wallet credit client (payout/refund)

## 개요
정산 시 wallet에 당첨금·환불을 반영하는 동기 HTTP 클라이언트. betting의 debit과 달리 **business-rejection 분기가 없다** — 모든 실패는 일시 장애로 보고 재시도/DLQ.

## 핵심 발견 — WON은 wallet credit **2-leg**
wallet의 `CreditCommand.Source` 문서를 정독하고서야 알았다:
- **WON**: ① locked에 잡아둔 stake를 `USER_LOCKED`로 반환(BET_REFUND) + ② house 이익을 `HOUSE_POOL`로 지급(BET_PAYOUT). **단일 credit으로는 double-entry가 깨진다**(house가 stake까지 과지급, 또는 locked가 영영 안 풀림).
- **PUSH/VOID**: stake를 `USER_LOCKED`로 환불(1-leg).
- **LOST**: wallet 호출 **없음**(접수 때 이미 available에서 차감, locked에 잡혀 있음).

이 발견이 SettlementService의 payout 분기를 결정했다(커밋 009).

## 작업 내역
- `WalletCreditClient.credit(idempotencyKey, userId, amount, source)` — `POST /internal/v1/wallet/transactions/credit`, `Idempotency-Key` 헤더 + `source`(USER_LOCKED/HOUSE_POOL).
- 실패 처리: 모든 non-2xx/timeout/connection → `DependencyUnavailableException`(resilience4j `walletClient` breaker가 기록하는 **유일한** 예외). 재시도 → 소진 시 DLQ. betting처럼 INSUFFICIENT_BALANCE 같은 business 분기가 없다(credit은 house가 지급, locked엔 환불할 stake가 이미 있음).
```java
@CircuitBreaker(name="walletClient", fallbackMethod="creditFallback")
public UUID credit(String idemKey, UUID userId, Money amount, CreditSource source) { ... }
```
- 멱등 키는 호출자 책임: betId 파생 키로 retry 시 wallet이 dedup.

## 결과
payout/refund를 안전하게 호출할 클라이언트. WireMock 단위 테스트(성공/5xx/timeout) 동봉.

## 요약
- 제네릭 credit(키+source), 모든 실패=DependencyUnavailableException.
- **WON=2-leg(USER_LOCKED stake 반환 + HOUSE_POOL 이익)**, PUSH/VOID=1-leg, LOST=무호출.

## 다음 작업
resolver+payout+outbox를 묶는 SettlementService.

## 핵심 확인
`WalletCreditClientTest` (WireMock, 동봉).

## 기억/설명 Level
- **L1**: credit 호출 형태, 실패=재시도/DLQ.
- **L2**: business 분기가 왜 없나.
- **L3 (외워서 설명)**: **WON 정산의 2-leg payout**(locked stake 반환 + house 이익) — wallet double-entry 모델에서 왜 필수인가. 면접 핵심.
